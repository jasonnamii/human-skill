# Human Skill — 인간 메커니즘 16축 엔진

인간이 왜 그렇게 반응하는지를 16축·44메타원리 체계로 구조화한 진단·예측·설계 도구. 각 축은 메커니즘→체인→자명명제(축의 최소 구성 단위)로 분해되며, 자명명제는 스포크 내부에서 체인의 근거로 참조된다.

---

## ⛔ 절대 규칙

| # | 규칙 | 이유 |
|---|---|---|
| 1 | **스포크 없이 기억으로 실행 금지** — 해당 축/메타원리 상세가 필요하면 반드시 `@uses` 레퍼런스 로드 | 16축×44메타원리의 내부 체인을 기억만으로 정확 재현 불가 |
| 2 | **스크리닝 없이 상세 진입 금지** — 반드시 §2 스크리닝 → 활성 축 식별 → 해당 스포크만 로드 | 전수 로드는 토큰 낭비 + 초점 분산 |
| 3 | **메타원리 무시 금지** — 복수 축 활성 시 반드시 메타원리로 축간 상호작용 해석 | 개별 축 나열은 단편적. 메타원리가 인과 구조를 결정 |

## §1. 모드 자동 판별

| 입력 유형 | 모드 | 판별 기준 |
|---|---|---|
| 관찰된 행동·현상 | **진단** | "왜 이런 행동이 발생했는가?" — 과거·현재의 인과 추적 |
| 대상의 상태·맥락 | **예측** | "이 조건에서 어떤 행동이 나타나는가?" — 미래 행동 확률 추정 |
| 원하는 행동·반응 | **설계** | "어떤 메커니즘을 활성화해야 하는가?" — 트리거 조건 역추적 |

**복합 시**: 진단→예측→설계 순서 파이프라인 가능. 모드 애매 시 형에게 확인.

## §3. 메타원리 라우팅 — 활성 축 조합 → 상호작용 해석

스크리닝에서 2개 이상 축이 활성 → **Python 라우터로 즉시 조회** (스포크 전체 로드 불필요).

### Python 라우팅 (권장 — 토큰 절약 + 정확도)

```bash
# 기본: 활성 축 → 메타원리 라우팅
python scripts/meta_router.py 축1 축5 축H

# 상세: 메타원리 + 활성 축의 자명명제·체인 의미·설계원칙 포함
python scripts/meta_router.py --detail 축1 축5

# 축 풀 상세: 특정 축의 체인 전문 출력 (스포크 로드 대체)
python scripts/meta_router.py --axis 축1
python scripts/meta_router.py --axis 축E --table  # 특수 테이블 포함

# 모드 프로토콜: 진단/예측/설계 실행 절차
python scripts/meta_router.py --mode 진단

# 유틸리티
python scripts/meta_router.py --loops   # 자기강화 루프만
python scripts/meta_router.py --hubs    # 허브 축 요약
python scripts/meta_router.py --all     # 전체 44개 매트릭스
```

**권장 워크플로우**: ①기본 라우팅으로 활성 축+메타원리 식별 → ②`--detail`로 자명명제·설계원칙 확인 → ③필요시 `--axis`로 체인 전문 조회. 이 순서면 스포크 .md 파일을 직접 로드할 필요가 거의 없다.

**폴백**: Python 실행 불가 시 `references/` 스포크 파일에서 수동 조회.

### 메타원리 요약 (44개 = 6계열: M6+CM8+SM6+EM8+EmoM8+BM8)

전체 목록: `python scripts/meta_router.py --all` | 루프만: `--loops` | 허브: `--hubs`

**자기강화 루프 6개**: M5, CM7, SM6, BM3, BM4, EmoM5
**설계 최우선 3**: SM6(상승/하강 분기) → EmoM5(조절-충족 루프) → BM3(습관-정체성)

### 저활용 메타원리 활성화 가이드

44개 중 ~10개만 반복 활성화, 나머지 34개는 저활용. → **`references/low-usage-meta-guide.md` 로드** (도메인 특화형 / 조합 전용형 / 극단 상황형 3분류 + 활성 트리거).

스크리닝 후 활성 축 3개+ → `--detail`로 메타원리 확인 시 3분류 조건 매칭 1회 체크.

## §4. 상세 로딩 — 스포크 라우팅

스크리닝에서 활성화된 축 영역에 따라 해당 스포크만 로드.

**Python 우선**: `--detail`이 자명명제·체인 의미·설계원칙을 포함하므로 대부분 스포크 로드 불필요. `--axis`는 체인 전문까지 출력하여 스포크를 완전 대체.

**폴백 스포크**: Python 실행 불가 시 — 축1~7→`axes-cognitive.md` | A~C→`axes-social.md` | D~E→`axes-motivation.md` | F~G→`axes-evolution.md` | H→`axes-emotion.md` | I→`axes-behavior.md`

## §5. 모드별 실행 요약

**진단**: 행동기술→스크리닝→체인추적→메타원리매핑→인과조합
**예측**: 상태파악→축식별→메타원리시뮬→허브파급→시나리오분기→**확률 근거 부착**
**설계**: 목표정의→역추적→트리거설계→시너지조합→리스크체크

### 예측 모드 확률 근거 필수 프로토콜

예측 모드에서 확률 수치 제시 시 3단계 필수: ①유사 선행 사례 1건+ 인용 ②기저율(base rate) 제시 ③확률에 근거 태그 부착(`[통계]` `[유추]` `[전문가판단]` `[모델추정]`). 근거 없는 확률 수치 제시 금지. 웹 검색 미확보 시 `[모델추정]` + "확인 필요" 표기.

상세 프로토콜: `python scripts/meta_router.py --mode 진단|예측|설계`

## §6. 축 체계 계층 구조

```
[기저층 — 진화적 설계 원리]
축G(위협관리) → 축1·3·5·6·C·H·I에 궁극적 원인 제공
축F(번식전략) → 축4·E에 궁극적 원인 제공

[인지 엔진 — 개인 내 정보 처리]
축1(인지자원) ← 기저 제약
  ↓ M1
축5(감정선행) ← 기본 판단 모드
  ↓
축2(속성대체) / 축3(손실회피) / 축4(스토리텔링) / 축6(기억왜곡)
축7(판단잡음) ← 독립 오류원

[감정 과정 — 허브①]
축H(감정구성·조절) ← 7축과 교차하는 중심 허브

[동기 엔진 — 행동의 "왜"]
축D(인과소재) ↔ 축E(욕구충족) ← SM6 쌍방향 강화 루프

[사회적 메커니즘 — 대인 간]
축A(호혜) ↔ 축B(자기상) ↔ 축C(정체성)

[행동 실행 — 허브②]
축I(행동루프·습관) ← 8축과 교차하는 실행 허브
```

**3대 허브 축**: 축H(감정, 8접합) · 축I(행동, 8접합) · 축E(동기, 6접합) — 상세: `python scripts/meta_router.py --hubs`

### 진화축(F·G) 자동 주석 규칙

축F/G 활성 시, 진단·예측 모드에서 1줄 주석 자동 삽입: "이 행동의 진화적 기원: [짝 선택·지위·위협 탐지 등] → 현대 맥락 불일치: [구체 서술]". 설계 모드에서는 "설계 레버 아님" 명시 + 근접 축(F→축4·E / G→축1·3·5·6·C·H·I)에서 설계.

## Gotchas

| # | 안티패턴 | 올바른 접근 |
|---|---|---|
| 1 | 16축 전체 상세 분석 시도 | 스크리닝 후 활성 축 3~5개에 집중 |
| 2 | 개별 축만 나열하고 메타원리 무시 | 복수 축 활성 시 반드시 메타원리로 상호작용 해석 |
| 3 | 축을 독립적으로 취급 | 축은 계층 구조(§6). 기저층→인지→감정→동기→사회→행동 흐름 존재 |
| 4 | 진화축(F·G)을 직접 설계 대상으로 취급 | 진화축 = 궁극적 설명 도구. 진단에서는 자동 주석 출력(§6 규칙), 설계 레버는 근접 축 |
| 5 | 허브 축(H·I·E) 무시 | 3대 허브가 다른 축을 연결. 허브 미고려 시 인과 구조 불완전 |
| 6 | 메타원리 방향 무시 (→ vs ↔) | 단방향(→)과 쌍방향(↔)은 설계 함의가 다름. ↔는 자기강화 루프 가능 |
| 7 | 자명명제를 직접 인용 | 축→체인→메타원리 수준에서 분석 |
